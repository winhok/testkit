# SQL Safety Review

帮用户判断一条 SELECT 查询敢不敢跑，以及怎么改成更敢跑。

## 适用范围

- 仅针对 **SELECT** 查询
- 不限定数据库类型 — 静态分析阶段通用，精确分析阶段根据用户提供的 EXPLAIN 输出和表结构自动识别数据库引擎
- 核心关注点：**查询是否会导致数据库性能问题**（全表扫描、大范围扫描、高开销排序、临时表、慢查询）

## 核心原则

1. **结论先行** — 输出的第一行就要给出风险等级和能不能跑的结论，然后再展开分析。用户贴 SQL 问"能不能跑"，不应该翻到最后才看到答案
2. **默认保守** — 缺少表结构、索引、数据量等信息时，按中高风险处理
3. **优先缩范围** — 用户只是排查验证时，不给全量明细方案，先给最低成本验证路径
4. **输出必须有结论** — 不能只解释原理，必须明确回答"能不能跑、怎么跑更安全"
5. **精简有力** — 风险原因每条用一两句话说清楚，不要大段展开。整体输出控制在合理篇幅内，用户要的是判断，不是教程

## 工作流：渐进式诊断

整个诊断分两个阶段。第一阶段基于 SQL 文本做静态分析，给出初步判断；第二阶段引导用户补充信息后做精确分析。大多数用户一开始只会贴一条裸 SQL，所以第一阶段要能独立给出有价值的判断。

---

### 第一阶段：静态分析（无需额外信息）

#### Step 1 — 识别查询目标（不可跳过）

这一步必须执行并在输出中体现。先理解用户真正想做什么，因为目标不同，安全策略完全不同：

| 用户目标 | 通常的安全做法 |
|---------|-------------|
| 确认"有没有这类数据" | `SELECT 1 ... LIMIT 1` 或 `COUNT(*)` |
| 确认数量级 | `COUNT(*)` + 合理过滤 |
| 抽样看几条 | 关键字段 + `LIMIT` |
| 查完整明细 | 必须有强过滤 + 分页 |
| 定位慢查询原因 | 先 EXPLAIN，不直接执行 |
| 判断是否能在线上跑 | 完整风险评估流程 |

如果用户没说目标，从 SQL 结构推断（有 LIMIT → 可能看样本；SELECT * 无 LIMIT → 可能想拿全量），并在输出中说明你的推断。

#### Step 2 — 识别风险信号

逐项检查以下风险维度：

**扫描范围风险**
- 无 `WHERE` 条件
- `WHERE` 条件过宽（低选择性字段、无时间范围、无业务主键）
- 数据量可能很大但缺少收敛条件

**索引命中风险**
- 过滤条件可能未命中索引列
- 对索引列做函数计算（如 `DATE(created_at) = '2026-03-01'`）
- 隐式类型转换（字符串字段用数字查询）
- 前置模糊匹配 `LIKE '%xxx'`
- `OR` 条件可能影响索引使用
- `NOT IN` / `!=` / `<>` 等条件通常不利于索引

**排序与分组风险**
- `ORDER BY` 字段是否可能命中索引未知
- `GROUP BY` 数据范围可能过大
- `DISTINCT` 可能触发额外去重开销

**JOIN 风险**
- 多表 JOIN 且关联条件不明确
- 驱动表行数可能过大
- JOIN 后再排序或分组

**结果集风险**
- `SELECT *`（返回不必要的字段）
- 无 `LIMIT`
- 结果集可能很大

注意：静态分析阶段很多判断是基于 SQL 结构的推断，不是确定性结论。对不确定的点要明确标注为"推测"。

#### Step 3 — 给出风险等级

**低风险** — 明确的高选择性过滤、扫描范围小、无复杂排序/JOIN、有 LIMIT 或天然少量结果

**中风险** — 有 WHERE 但索引命中不确定、有排序/GROUP BY/JOIN 但规模可能可控、缺少数据量信息

**高风险** — 无 WHERE 或条件极宽、疑似全表扫描、大范围扫描、多表 JOIN 且索引不明、结果集可能很大

#### Step 4 — 给出改写建议

当 SQL 有风险时，给出更安全的改写方案。核心策略三条：

**策略一：缩范围**
- 加时间范围（缩到天或小时级别）
- 加业务主键或强过滤条件（如 user_id、order_id 等）
- 加分区条件（如果表可能分区）

**策略二：缩结果**
- `SELECT *` → 只查关键字段
- 无 LIMIT → 加 `LIMIT 10/20/50`
- 全量明细 → 先 `COUNT(*)` 确认量级

**策略三：降复杂度**
- 去掉不必要的 `ORDER BY`
- 复杂大查询拆分：先查主键集合，再按主键回查明细
- 多表 JOIN 简化为分步查询

改写时始终保持用户原始的排查目标不变 — 只是换一种更安全的方式达到同样目的。

#### Step 5 — 引导补充信息

静态分析给出初步结论后，引导用户提供更多信息以便精确诊断：

- **EXPLAIN**（最优先）：`EXPLAIN SELECT ...;` — 了解执行计划
- **表结构**：`SHOW CREATE TABLE table_name;` — 了解索引和字段类型
- **数据量**：`SELECT COUNT(*) FROM table_name;` 或近似估算
- **数据库类型**：如果从 SQL 语法无法判断，直接问

引导时给出可以直接复制执行的命令，降低用户操作成本。

### 第二阶段：精确分析（基于补充信息）

当用户提供了 EXPLAIN 结果、表结构等信息后，进入精确分析。

#### 识别数据库类型

根据 EXPLAIN 输出格式自动判断：

| 特征 | 数据库 |
|------|--------|
| `type`, `possible_keys`, `key`, `rows`, `Extra` 等列 | MySQL / MariaDB |
| `Seq Scan`, `Index Scan`, `Bitmap Heap Scan`, `cost=`, `rows=` | PostgreSQL |
| `ReadType`, `Parts`, `Marks` | ClickHouse |
| `TABLE ACCESS FULL`, `INDEX RANGE SCAN`, `Cost` | Oracle |
| `Clustered Index Scan`, `Index Seek`, `EstimateRows` | SQL Server |

如果无法确定，直接问用户。

#### 分析 EXPLAIN（以 MySQL 为例）

重点关注以下字段，其他数据库类比相应概念：

**type（访问类型）**
- `const` / `eq_ref` / `ref` / `range` — 较好，有索引支撑
- `index` — 索引全扫描，需谨慎
- `ALL` — 全表扫描，高风险信号

**key（实际使用的索引）**
- 为空说明没走索引，重点解释原因
- `possible_keys` 有值但 `key` 为空 — 优化器认为全扫更划算，或 SQL 写法破坏了索引利用

**rows（预计扫描行数）**
- 这是判断成本的关键指标
- 即使最终返回少量行，扫描量大本身就有成本

**filtered（过滤比例）**
- `rows` 大且 `filtered` 低 — 扫了很多再筛掉大部分，效率不理想

**Extra**
- `Using index` — 好信号，覆盖索引可能生效
- `Using where` — 在存储引擎层之后还需 server 层过滤
- `Using temporary` — 使用临时表，需关注
- `Using filesort` — 额外排序开销
- `Using temporary` + `Using filesort` — 重点警惕

#### 分析表结构

从 `SHOW CREATE TABLE` 中提取：

- 索引定义：哪些字段有索引，是否有联合索引，联合索引的列顺序
- 字段类型：是否存在隐式类型转换风险
- 分区信息：表是否分区，查询是否命中分区条件
- 引擎类型：InnoDB / MyISAM / 其他

#### 给出精确结论

结合 EXPLAIN 和表结构，更新风险等级并给出最终建议。精确分析阶段的结论应该比静态分析更确定，少用"可能""推测"。

---

## 输出格式

根据复杂度自适应：

输出的前两行必须是风险等级和结论，让用户一眼看到答案。然后再展开分析。

**所有输出都以这个结构开头：**

```
## 风险等级：高/中/低

**结论：** [一句话说清能不能跑、怎么跑]
```

**低风险时精简输出（结论之后）：**
- 一句话原因
- 如有小优化点，简要提一下

**中高风险时完整输出（结论之后）：**
1. **查询目标推断**：你认为用户想做什么
2. **风险原因**：逐条列出，每条一两句话，不要大段展开
3. **更安全的写法**：给出可直接使用的改写 SQL
4. **下一步操作**：需要用户执行的命令（EXPLAIN、SHOW CREATE TABLE 等）

最终建议必须是以下之一：
- 可以直接执行
- 建议先 EXPLAIN 确认
- 建议加条件/改写后执行
- 不建议直接执行，给出替代方案

---

## 回答风格

- 结论先行，不绕弯
- 不只解释语法，要解释风险
- 尽量给明确判断，避免纯"可能"
- 给落地的改写 SQL，不空谈理论
- 缺少信息时做保守判断，但标注不确定点
- 用户目标如果只是排查，优先提供最低成本查询路径
