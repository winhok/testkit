# log-analysis：服务端日志智能分析

## 职责

将一段混杂的服务端日志**拆解为可读的链路视图**，还原每条请求/任务的完整生命周期，识别异常、性能瓶颈和潜在风险，产出结构化的 Markdown 分析报告。

核心价值：把人工需要 30 分钟才能理清的日志，在几秒内变成一份清晰的诊断报告。

## 输入方式

支持两种输入，自动识别：

1. **直接粘贴** — 用户在对话中直接粘贴日志文本
2. **文件路径** — 用户提供日志文件的本地路径（如 `/data/services/xxx/logs/app.log`），可以是完整文件或指定行范围

如果用户提供的是远程服务器路径（如 `ec2-user@host:/path/to/log`），提示用户先把日志内容复制到本地或直接粘贴。

## 分析模式：三档自动切换

先判断用户意图，再结合日志量选择模式。这一步至关重要——用户问「推送为什么失败」和「帮我分析下这段日志」需要的是完全不同的报告。

### 聚焦模式（用户带着具体问题来的）

**触发条件**：用户的输入中包含明确的问题或排查目标，例如：
- 「推送为什么成功数是 0？」
- 「这个接口为什么慢？」
- 「登录报错了帮我看看」
- 「xxx 为什么失败了」

**核心原则**：直奔用户的问题。不要做全景分析。

**行为**：
1. 仍然拆分所有链路（需要全局视角才能找到问题），但**不展示**无关链路的详情
2. 在报告开头直接回答用户的问题（一段话结论）
3. 只展开与问题相关的链路详情、SQL 分析、服务调用分析
4. 其他链路只在「链路总览」表格中一行带过，标注「正常，与当前问题无关」
5. 深挖命令聚焦于用户的问题方向

**产出结构**：见后文「聚焦模式报告模板」

### 快速概览模式（日志量小，无具体问题）

**触发条件**：用户没有具体问题 + 日志 < 100 行或独立链路 ≤ 2 条

**核心原则**：简洁。10 行日志不需要 100 行报告。

**行为**：
1. 产出精简报告，控制在日志行数的 3-5 倍以内
2. 只输出：概览信息 + 链路总览表 + 问题（如有）+ 一句话结论
3. 不需要展开链路详情（日志量小，用户自己能看懂）
4. 如果完全没有问题，一句话结论即可，不要硬凑内容
5. 深挖命令只在发现可疑点时才给，正常情况省略

**产出结构**：见后文「快速概览报告模板」

### 深度诊断模式（日志量大，无具体问题）

**触发条件**：用户没有具体问题 + 日志 ≥ 100 行或独立链路 > 2 条

**行为**：完整分析所有链路，产出完整报告，包含：
- SQL 查询分析（慢查询、N+1 问题、全表扫描风险）
- 服务间调用链（Feign/HTTP/RPC 调用拓扑）
- 业务逻辑诊断（状态流转、成功率、重试模式）
- 性能瓶颈定位（耗时分布、批量操作效率）
- 安全风险扫描（敏感信息泄露、认证异常）

**产出结构**：见后文「深度诊断报告模板」

### 手动覆盖

用户可以通过 `--deep` 强制深度模式，`--quick` 强制快速模式。聚焦模式无需手动触发，由用户是否带具体问题自动判定。

## 执行步骤

### 第一步：获取日志

- 粘贴文本 → 直接使用
- 文件路径 → 读取文件内容（支持 `sed -n 'start,end p'` 形式的行范围提示）
- 如果日志内容为空或无法读取，提示用户检查路径

### 第二步：识别日志格式

自动识别日志格式。常见的 Java/Spring Boot 格式：

```
{timestamp} [{thread}] - [{level}] [{logger}] traceId:{traceId} - {message}
```

关键字段提取：
- **时间戳**：精确到毫秒
- **线程名**：用于区分并发活动
- **日志级别**：DEBUG / INFO / WARN / ERROR
- **Logger 类名**：定位代码位置
- **TraceId**：串联同一请求的所有日志（可能为空）
- **消息体**：包含 SQL、参数、响应等业务信息

如果格式不匹配预期，尝试以下策略：
1. 按时间戳 + 线程名分组（大部分日志都有这两个字段）
2. 如果连时间戳都没有，按连续行分组
3. 告知用户识别到的格式，确认是否正确

### 第三步：链路拆分

这是分析的核心环节。目标是把交错的日志行拆分为独立的逻辑链路。

**分组优先级**：
1. **TraceId 分组**（最可靠）— 相同 traceId 的行归为一组
2. **线程名分组**（次优）— 无 traceId 时，按线程名归组
3. **时间窗口 + 上下文推断**（兜底）— 相邻时间内、语义相关的行归为一组

**注意事项**：
- 同一个线程可能先后处理多个请求（线程池复用），需结合时间间隔判断
- 异步线程（如 `xxxAsyncExecutor-N`）通常缺少 traceId，但可以通过业务 ID 关联
- 定时任务线程（如 `xxl-job, JobThread-N`）独立成组
- HTTP 请求线程（如 `http-nio-xxx-exec-N`）通过 Request/Response 日志确定边界

### 第四步：逐链路分析

对每条链路执行以下分析：

#### 4.1 基础信息提取

- 链路类型：HTTP 请求 / 异步任务 / 定时任务 / 消息消费 / 其他
- 起止时间与总耗时
- 涉及的主要操作（SQL、外部调用、业务处理）

#### 4.2 异常检测

按严重程度分级：

| 级别 | 说明 | 示例 |
|------|------|------|
| CRITICAL | 业务功能失败 | 推送成功数为 0、支付失败、数据丢失 |
| WARNING | 潜在问题 | 上下文获取失败、重试次数多、耗时异常 |
| INFO | 值得关注 | 返回空结果、降级处理、配置异常 |
| SECURITY | 安全风险 | 日志中输出敏感信息（密钥、手机号、密码） |

#### 4.3 SQL 分析（深度模式）

从 MyBatis/Hibernate 日志中提取 SQL 语句和参数：
- **N+1 问题**：循环中反复执行相同结构的查询（参数递增）
- **全表扫描风险**：SELECT * 且无 WHERE 或条件过于宽泛
- **批量效率**：是否应该用 IN 查询替代循环单查
- **分页合理性**：游标分页的步长是否合适

#### 4.4 服务调用分析（深度模式）

识别 Feign、RestTemplate、WebClient 等服务间调用：
- 调用目标和频次
- 调用耗时
- 是否有重复调用或不必要的调用

#### 4.5 业务逻辑诊断（深度模式）

从日志中还原业务流程：
- 操作的起始状态和结束状态
- 关键分支走向
- 成功/失败统计
- 批量操作的进度和效率

### 第五步：生成报告

按产出结构输出 Markdown 报告。

### 第六步：深挖建议（交互式排查）

报告生成后，对于**无法仅凭当前日志确认根因**的问题，主动提供下一步排查命令，帮助用户获取更多上下文。

**命令生成原则**：
- 优先提供 `grep` 命令，因为这是服务器上最通用的工具
- 命令必须可直接复制执行，包含完整路径提示
- 每个命令附带一句话说明「为什么要查这个」
- 按排查优先级排序，最可能找到答案的命令排在前面

**常见排查场景及命令模板**：

1. **追踪同一 traceId 的完整日志**（当前片段可能截断了上下文）：
   ```bash
   grep '{traceId}' /path/to/app.log
   ```

2. **查看某个时间段前后的 ERROR/异常**：
   ```bash
   grep -n 'ERROR\|Exception\|Caused by' /path/to/app.log | grep '2026-03-09 21:1'
   ```

3. **追踪某个业务 ID 的全链路**（如订单号、推送操作 ID）：
   ```bash
   grep '{businessId}' /path/to/app.log
   ```

4. **查看某个线程在更大时间范围内的行为**：
   ```bash
   grep '\[{threadName}\]' /path/to/app.log | grep '2026-03-09 21:1'
   ```

5. **统计某类操作的频次**：
   ```bash
   grep '{pattern}' /path/to/app.log | wc -l
   ```

6. **查看其他关联服务的日志**（Feign 调用的下游服务）：
   ```bash
   grep '{keyword}' /path/to/downstream-service.log
   ```

**交互循环**：当用户执行完 grep 并将结果粘贴回来时，将新日志与之前的分析合并，更新诊断结论，必要时提供进一步的 grep 命令。这个「分析 → grep → 再分析」的循环可以持续多轮，直到定位到根因。

## 分析原则

### 聚焦问题，不做流水账

不要逐行翻译日志。重点是：
- **哪里出了问题**（或可能出问题）
- **为什么是问题**（影响是什么）
- **建议怎么做**（如何修复或进一步排查）

### 保持链路视角

始终以「一条完整的请求/任务从进入到完成经历了什么」为叙事主线。不要按日志级别或类名分组 — 那样会打断链路的连贯性。

### 量化优于定性

- 说「查询执行了 17 次，每次约 4ms，总计 68ms」而不是「查询执行了很多次」
- 说「批量推送 5012 用户，成功 0，耗时 1438ms」而不是「推送似乎失败了」

### 敏感信息提醒

分析过程中如果发现日志中包含敏感信息（API Key、手机号、密码、Token 等），在报告中标注为安全风险，建议做日志脱敏处理。报告本身直接引用原文即可（内部使用场景）。

## 边界情况处理

- **日志量极大**（> 5000 行）：提示用户缩小范围，建议用 `grep` 过滤特定 traceId 或时间段后再分析
- **日志格式不标准**：尽力解析，在报告开头说明格式识别结果，标注可能的解析误差
- **日志中无明显问题**：仍然产出报告，说明链路正常，给出链路摘要和性能基线数据
- **多种日志格式混合**：分别识别，在报告中标注不同格式的行数占比
