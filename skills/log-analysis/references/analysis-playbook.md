## 格式识别

优先提取这些字段：

| 字段 | 常见名称 | 用途 |
|------|----------|------|
| 时间 | timestamp/time/@timestamp | 排序、耗时计算、时间窗过滤 |
| 观测时间 | observedTimestamp/ObservedTimestamp | 判断采集延迟、跨端上报延迟 |
| 执行上下文 | thread/coroutine/worker/task/partition | 并发链路拆分 |
| 级别 | level/severity/SeverityText/SeverityNumber/status | 初筛异常和告警 |
| 组件 | logger/module/service/service.name/handler | 定位代码或服务边界 |
| 版本环境 | env/environment/service.version/deployment.environment | 判断灰度、版本差异和环境查错 |
| 关联 ID | traceId/trace.id/TraceId/correlationId/requestId/spanId/SpanId | 跨行、跨服务串联 |
| 消息体 | message/msg/event | 业务事实和异常内容 |

格式不统一时，分别识别不同格式并说明占比。无法解析的行不要丢弃，放入“未归组日志”。

结构化日志用字段语义解析，不要把整个 JSON 当纯文本。多行堆栈要合并成同一事件或标注可能被采集器拆散。

## 链路拆分

按可靠性从高到低分组：

1. 相同 trace/correlation/request/span id。
2. 相同线程、协程、worker、consumer partition，并结合时间间隔切分。
3. 相同业务实体、消息 id、任务 id、批次 id。
4. 时间窗口 + 语义连续性。

输出时标注归组依据。如果同一线程可能复用处理多个请求，必须用时间间隔、入口/出口日志或业务 id 再切分。

## 严重程度分级

| 级别 | 判定标准 |
|------|----------|
| CRITICAL | 功能失败、数据写错/丢失、任务整体失败、支付/订单/鉴权等关键路径失败 |
| WARNING | 超时边缘、重试过多、降级、局部失败、慢依赖、日志上下文缺失 |
| INFO | 值得关注但未造成明显失败，如空结果、跳过处理、配置分支 |
| SECURITY | 明文 token、密钥、密码、身份证件、手机号、敏感 header、越权/鉴权异常 |

不要只看日志级别。`INFO` 中可能包含业务失败，`ERROR` 也可能是已捕获且成功兜底。

## 性能诊断

优先回答：

- 总耗时从哪里开始到哪里结束？
- 最大耗时段是哪一段？是否有慢 SQL、慢 HTTP/RPC、队列等待、锁等待、重试退避？
- 是否存在重复调用或 N+1 查询？重复次数、单次耗时、累计耗时是多少？
- 批处理是否有批量大小、分页、并发度、成功/失败统计？

表达必须量化：写“接口总耗时 5.5s，其中下游调用占 5.1s”，不要只写“下游较慢”。

## 数据访问诊断

检查：

- SQL 或查询条件是否过宽、缺少索引线索、全量扫描风险。
- 循环单查是否可合并为批量查询。
- 写入后立即读取是否可能读到旧缓存或主从延迟。
- 查询的数据源、租户、环境、用户主体是否和用户预期一致。

没有执行计划时，不要断言索引一定失效；写成“存在全表扫描风险，需要用 explain 或慢查询详情验证”。

## 依赖与异步诊断

检查：

- 外部依赖的状态码、错误码、响应体摘要、耗时和重试次数。
- 消息消费是否重复、乱序、积压、ack/nack、死信或跳过。
- 定时/后台任务是否有任务开始、分片、进度、结束和统计日志。
- 异步链路缺 trace 时，是否能用业务 id 或消息 id 串联。

## 日志质量与安全风险

标注这些问题：

- 缺少 trace/correlation id，导致跨服务无法串联。
- trace/span/service/env/version 字段不一致，导致 logs、metrics、traces 不能互相跳转。
- 入口或出口未打印关键字段，导致无法证明字段是否传递。
- 同一字段多种命名但没有映射说明。
- 输出敏感信息，如凭证、token、密钥、手机号、证件号、完整地址。
- 大对象、完整请求体或响应体噪音过多，遮蔽关键事件。
- 把 requestId、traceId、userId、orderId、pod instance 等高基数字段作为 Loki label 或长期高成本索引。
- 低价值 debug/心跳日志量过大，真正异常被噪音遮蔽；适合转 metrics 的周期性计数仍长期作为日志保存。

## 边界情况

- 日志片段缺开头或结尾：只分析可见范围，并提示补齐 trace 或时间窗。
- 多种日志格式混合：分格式处理，不要强行套一个解析规则。
- 时钟不同步：若跨服务时间倒序，标注可能的时钟偏差。
- 没有明显问题：给出正常链路摘要、耗时基线和“未发现异常”的边界。
