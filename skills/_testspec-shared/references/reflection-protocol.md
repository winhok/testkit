# TestSpec 反思协议（Reflection Protocol）

> 通用反思迭代框架。在核心产物首次生成后，通过结构化自问自答发现问题并修正，替代静态自检清单。

## 目录

- 触发时机
- 3 步反思
- 质量门槛
- 迭代规则
- 适用边界

## 触发时机

- **必须触发**：核心产物首次生成后（每个 skill 至少 1 轮反思）
- **自动触发**：反思发现质量不达标时，自动进入迭代
- **响应式触发**：用户对产物提出反馈后

---

## 3 步反思

### Step 1：完整性自问

> "我是否遗漏了重要内容？"

按当前 skill 的职责自查：

**analysis：**
- 是否有功能模块被跳过？
- 风险点是否仅停留在表面，没有深入分析？
- 待澄清项是否只是把需求原文复述了一遍？

**points：**
- 是否有 analysis 中识别的风险点没有对应测试点？
- 各模块的测试点数量是否均衡？（某个模块只有 1-2 个点，其他有 10+ 个）
- 是否遗漏了异常/边界/集成类别的测试点？

**generate：**
- 是否所有测试点都有对应用例？（覆盖率 ≥ 95%）
- 冒烟用例是否全部为 P1？数量是否与业务复杂度匹配？（过多或过少都是异常信号）
- P1/P2/P3 分布是否合理？（纯 P1 或纯 P3 都是异常信号，但不套固定比例）
- 是否遗漏了关键的异常和边界场景？

**review：**
- 14 项检查是否都执行了？（R1-R6 + H1-H8）
- 问题是否有具体的用例 ID 和位置？
- 整改建议是否具体到可执行？

### Step 2：质量自评

> "产出是否有价值，还是在走形式？"

- **复述性检查**：产出中有多少内容是直接从输入材料复述的？复述占比 > 30% 说明分析不够深入
- **模板化检查**：产出是否像是套模板填空？每个问题/测试点/用例是否有针对性？
- **具体性检查**：结论、建议、验证要点是否足够具体？能否直接指导下一步行动？
- **冗余检查**：是否存在重复表达同一意思的内容？

### Step 3：迭代决策

> "消费者拿到这个产物，能直接用吗？"

消费者定义：
- analysis 的消费者 → points 作者（需要从中提炼测试点）
- points 的消费者 → generate 作者（需要从中展开用例）
- generate 的消费者 → 执行测试的人（需要按步骤执行并判断通过/失败）
- review 的消费者 → generate 作者（需要按建议修正用例）

判断标准：
- **可直接使用** → 完成反思，进入下一步
- **需要修正** → 标记具体修正项，进入迭代
- **需要大幅返工** → 考虑回溯上游（参见 context-protocol.md 的回溯机制）

## 质量门槛（按 skill 差异化）

### analysis 质量门槛

- 无复述性内容（不把需求原文换个说法重新写一遍）
- 每个问题必须指出"为什么这是风险"，而非仅描述现象
- 待澄清项必须是真正需要确认的问题，不是显而易见的答案

### points 质量门槛

- 各模块至少覆盖 Functional + 1 个其他类别（Boundary / Exception / Integration / Non-Functional）
- 每个测试点的验证要点必须可独立验证
- 测试点间无语义重复

### generate 质量门槛

- 测试点覆盖率 ≥ 95%
- 冒烟用例全部为 P1，数量由 Agent 根据业务复杂度自主判断
- 优先级分布由 Agent 根据风险判断，不设固定比例（但须合理——纯 P1 或纯 P3 都是异常信号）
- 每条用例的步骤和预期结果具体到可直接执行
- 无模板化填充（如"正常显示""操作成功"等模糊预期）

### review 质量门槛

- 每个问题必须有量化指标（覆盖率百分比、违规数量等）
- 每个问题必须有具体证据（用例 ID、字段内容等）
- 整改建议必须可执行（不能只说"需要改进"）

## 迭代规则

- **最多 2 轮迭代**：防止过度优化，2 轮后如仍不达标，标注剩余问题交给用户
- **迭代范围最小化**：只修正反思中发现的具体问题，不做全面重构
- **迭代可见性**：告知用户本次经过了几轮迭代、修正了什么

### 迭代流程

```
生成产物 v1
  → 反思 Round 1（完整性 + 质量 + 决策）
  → 若需修正 → 修正产物 → 产物 v2
  → 反思 Round 2（仅检查 Round 1 发现的问题是否修正）
  → 若仍有问题 → 标注剩余问题，交付产物 v2 + 问题清单
  → 告知用户：迭代次数 + 修正摘要
```

---

## 适用边界

- 本协议适用于所有 testspec-* skill 的产物生成后环节
- 反思是内化过程，不需要输出完整反思日志（除非用户要求）
- 只输出修正摘要和迭代次数
