# TestSpec 测试类型策略单一数据源

> 目标：为 `testspec-generate` 提供稳定的测试类型策略映射。此文件定义“按什么策略展开用例”，不定义最终导出格式。

## 使用原则

- `testspec-generate` 必须先识别用户指定的测试类型，再读取对应策略。
- 未指定测试类型时，默认使用 `functional`。
- 用户可指定多个测试类型；生成时按组合策略处理，但最终产物格式不变。
- 本文件只约束覆盖重点、设计技术和粒度，不改变 `testcases.json`、Excel、XMind 的字段结构。

## smoke

- 名称：冒烟测试
- 目标：最小化验证核心主流程是否可用
- 推理触发信号：
  - 用户提到"快速验证""冒烟""基本可用"
  - 材料中有明确的核心业务闭环
  - 上线前紧急测试场景
- 设计策略：
  - 每个核心功能点最多展开 1-2 条用例
  - 优先覆盖最短业务闭环
  - 只保留阻塞性强、失败影响大的场景
- 推荐技术：
  - 主流程场景法
  - 关键节点错误推测
- 输出倾向：
  - 优先生成 `type=冒烟`
  - 必须为 `P1`

## functional

- 名称：功能测试
- 目标：验证功能正确性与主要业务路径
- 推理触发信号：
  - 未指定具体测试类型（默认策略）
  - 用户提到"功能测试""验证功能""完整测试"
  - 新功能上线需要全面验证
- 设计策略：
  - 覆盖有效等价类与关键无效等价类
  - 优先覆盖主路径、核心分支、关键规则
  - 不为追求全面而无限膨胀用例数
- 推荐技术：
  - 等价类划分
  - 场景法
  - 判定表
- 输出倾向：
  - 优先生成 `type=正向`，必要时补充 `负向`

## boundary

- 名称：边界测试
- 目标：验证数值、长度、容量、日期时间等边界条件处理
- 推理触发信号：
  - 材料中有明确的数值范围、长度限制、容量上限
  - 涉及表单输入、文件上传、数据导入等有约束的场景
  - 上游分析标注了边界值相关的风险
- 设计策略：
  - 优先覆盖边界值、边界值±1、空值、满值、典型值
  - 同一验证意图的数据点可以拆成多个独立用例
- 推荐技术：
  - 边界值分析
  - 等价类划分
- 输出倾向：
  - 优先生成 `type=边界`

## exception

- 名称：异常测试
- 目标：验证系统异常处理能力和错误反馈质量
- 推理触发信号：
  - 涉及网络请求、外部服务调用
  - 有支付、数据修改等需要事务一致性的操作
  - 上游分析标注了异常处理缺失的风险
- 设计策略：
  - 覆盖异常输入、重复提交、超时、网络异常、服务异常、并发冲突
  - 关注错误提示、回滚、幂等、数据一致性
- 推荐技术：
  - 错误推测
  - 异常场景法
  - 状态迁移
- 输出倾向：
  - 根据场景落到 `type=负向` 或 `type=异常`

## permission

- 名称：权限测试
- 目标：验证角色访问控制、功能入口暴露和数据隔离
- 推理触发信号：
  - 材料中有角色定义或权限矩阵
  - 涉及多租户、数据隔离
  - 上游分析标注了权限交叉或越权风险
- 设计策略：
  - 明确角色矩阵
  - 同时覆盖有权限、无权限、越权三类场景
  - 优先覆盖高风险操作和敏感数据访问
- 推荐技术：
  - 权限矩阵
  - 场景法
- 输出倾向：
  - 通常落到 `type=负向` 或 `type=异常`

## security

- 名称：安全测试
- 目标：验证基础安全防护机制是否缺失
- 推理触发信号：
  - 涉及用户认证、会话管理、密码处理
  - 有用户输入直接展示的场景（XSS 风险）
  - 涉及敏感数据传输或存储
- 设计策略：
  - 聚焦输入校验、输出编码、认证鉴权、敏感信息保护、会话管理
  - 只覆盖需求材料中能合理推导出的安全场景
- 推荐技术：
  - 错误推测
  - 风险驱动场景法
- 输出倾向：
  - 常落到 `type=异常` 或 `type=负向`

## compatibility

- 名称：兼容性测试
- 目标：验证系统在不同设备、浏览器、系统或版本中的表现一致性
- 推理触发信号：
  - 材料中指定了目标浏览器或设备列表
  - 涉及响应式布局、移动端适配
  - 有新旧版本并行或数据迁移场景
- 设计策略：
  - 先从需求中提取目标环境矩阵
  - 同一功能在不同环境中生成独立用例
  - 关注显示、交互、上传下载、输入法、布局、版本差异
- 推荐技术：
  - 环境矩阵
  - 场景法
- 输出倾向：
  - `type` 可保持业务类型；重点通过 `feature/title/steps` 区分环境

## 多类型组合规则

- `functional + boundary`：先保证正向主路径，再补关键边界
- `functional + exception`：先保主路径，再覆盖关键失败路径
- `smoke + functional`：冒烟只保最小闭环；其余核心场景落功能测试
- `permission + security`：先覆盖权限矩阵，再补高风险安全场景

## 默认回退

- 未识别类型时，回退为 `functional`
