# TestSpec 思考协议（Thinking Protocol）

> 通用推理框架，替代所有 testspec-* skill 中的 if-else 路由。每个 skill 在做决策前，必须走完三个推理阶段，而非用关键词匹配选择策略。

## 目录

- 使用方式
- Phase 1：材料评估
- Phase 2：策略推理
- Phase 3：执行决策
- 工具使用决策
- Phase 4：执行规划
- 适用边界

## 使用方式

在 skill 执行流程的「策略选择」环节，调用本协议。输出推理过程和执行决策摘要，然后再进入实际执行。

---

## Phase 1：材料评估

在做任何决策之前，先回答以下 5 个引导问题：

### 1.1 我拿到了什么？

- 列出所有可用输入（proposal.md、requirements.md、requirements-analysis.md、testpoints.md、testcases.json、外部链接、代码仓库等）
- 标注每个输入的存在状态（存在/缺失/部分）

### 1.2 信息密度如何？

- **完整 PRD**：有详细功能描述、交互流程、异常处理、数据规则
- **简短描述**：只有功能名称和简单说明，需要推断细节
- **代码仓库/接口文档**：需要从技术实现反推业务需求
- **混合材料**：多种来源，需要交叉验证

### 1.3 有哪些显式信号？

- 用户明确指定的模式、策略、关注点
- 文档中标注的优先级、风险、约束
- 上游 skill 产物中的上下文元数据（参见 context-protocol.md）

### 1.4 有哪些隐式信号？

- 材料中频繁出现的模式（TBD、待定、后续确认 → 完整性风险）
- 技术方案的复杂度（第三方依赖、并发、分布式 → 可行性关注）
- 业务流程的状态转换密度（多状态、多角色 → 逻辑复杂度高）
- 安全敏感场景（支付、认证、用户数据 → 安全关注）
- 材料中的矛盾或歧义（同一概念不同命名 → 一致性问题）

### 1.5 这些信号对本步骤意味着什么？

- 综合 1.1-1.4 的结果，判断对当前 skill 执行的影响
- 识别信息缺口和风险区域

---

## Phase 2：策略推理

基于 Phase 1 的评估结果，推导策略组合。不查关键词表，而是通过 3 个核心问题引导推理：

### 2.1 核心问题

根据当前 skill 的职责，回答以下问题：

**对 analysis（需求分析）：**
- "这份材料最大的测试风险在哪里？" → 推导分析模式
- "材料中是否有技术/安全/歧义/矛盾特征？" → 推导补充模式
- "用户的实际关注点是什么？" → 调整模式权重

**对 points（测试点）：**
- "核心业务价值链是什么？" → 决定覆盖深度
- "哪些区域风险最高？" → 决定 P1 分配
- "验证粒度应该多细？" → 控制测试点数量

**对 generate（用例生成）：**
- "用户需要什么粒度的测试？" → 推导测试类型策略
- "哪些场景最容易被遗漏？" → 补充覆盖
- "冒烟用例应该覆盖哪条业务主线？" → 冒烟用例选择

**对 review（用例评审）：**
- "这批用例最可能出什么问题？" → 聚焦检查重点
- "上游分析发现了哪些风险？" → 对风险区域加深检查
- "用例规模和复杂度如何？" → 校准检查深度

### 2.2 策略推导

- 从核心问题的回答中，推导出策略组合
- 策略必须从对应 skill 的数据源中选取（`testspec-analysis/references/analysis-modes.md` / `testspec-generate/references/test-type-strategies.md`）
- 但选择理由来自推理，而非关键词匹配

---

## Phase 3：执行决策

输出执行计划摘要，包含以下要素：

### 3.1 决策记录

```
策略选择：<选了什么模式/策略/类型>
选择理由：<基于哪些信号推导出这个选择>
预期规模：<产出的大致规模（问题数/测试点数/用例数）>
风险点：<可能不准确或遗漏的地方>
```

### 3.2 向用户简要说明

用 1-2 句话告知用户：选择了什么策略、为什么这样选择。不需要展示完整推理过程。

## 工具使用决策

在 Phase 1 材料评估中，如果发现信息不足，按以下优先级获取外部信息：

1. **本地文件**：项目内的代码文件、文档文件（读取文件或匹配路径）
2. **用户链接**：proposal.md 或用户消息中提到的链接（抓取可访问内容）
3. **代码仓库**：相关的源代码（搜索文本并读取关键文件）
4. **搜索引擎**：公开的技术文档或标准（按当前执行环境可用能力搜索）

### 决策原则

- 信息足够时不获取额外信息
- 只获取对当前决策有帮助的信息
- 获取后将关键信息纳入 Phase 1 评估
- 工具使用是可选的，不是必须的

## Phase 4：执行规划（Agentic 决策层）

> 从"推理出策略"升级为"规划完整的执行方案"——Agent 不只决定做什么，还要决定用什么工具、怎么验证、何时求助。

### 4.1 工具与资源选择

基于 Phase 1-3 的结论，决定执行所需的工具链：

- **需要解析文档吗？** 用户提供了 PDF/DOCX → 调用可用解析脚本；纯文本 → 直接读取
- **需要搜索代码仓库吗？** 需求涉及现有功能改造 → 搜索相关代码并读取关键文件
- **需要查阅外部文档吗？** 涉及第三方 API、标准规范 → 使用当前环境可用的抓取或搜索能力
- **需要调用校验脚本吗？** 产出 testcases.json 后 → 调用 `validate_testcases.py` 自检

### 4.2 自检策略

产出核心产物后，Agent 自主验证质量：

- **格式校验**（generate 产出后）：运行 `python skills/_testspec-shared/scripts/validate_testcases.py --input testcases.json --testpoints specs/testpoints.md --pretty`
- **覆盖度检查**：从校验结果的 `coverage` 字段判断是否需要补充用例
- **分布检查**：从校验结果的 `distribution.warnings` 判断比例是否异常
- **自动修复**：若发现 errors → 修复后重新生成 → 再次自检（最多 2 轮）
- **反模式对照**：生成产物后，对照本 skill 的「反模式识别」段落自查

### 4.3 人类介入判断

以下情况**暂停执行，向用户确认**：

- **材料质量 < 中等**：信号密度低、TBD 过多 → 先向用户确认理解是否正确
- **发现需求矛盾**：同一规则在不同位置定义冲突 → 列出矛盾点请用户澄清
- **自检 2 轮仍有 error 级问题** → 暂停并报告，让用户决定是否继续
- **覆盖率 < 80%** → 可能是上游测试点不足，建议回退到 testspec-points

### 4.4 执行计划记录

```
工具链：<需要调用的工具/脚本>
自检计划：<产出后用什么方式验证>
人类介入条件：<在什么情况下暂停>
预期产出：<文件路径和格式>
```

---

## 适用边界

- 本协议适用于所有 testspec-* skill 的策略决策环节
- 用户明确指定策略时，跳过 Phase 2 推理，直接进入 Phase 3
- 推理过程应内化，不输出完整推理日志给用户（除非用户要求）
- 推理结果通过 context-protocol.md 传播给下游 skill
- Phase 4 的执行计划是 Agent 内部决策，只向用户输出关键结论（选了什么工具、为什么暂停）
