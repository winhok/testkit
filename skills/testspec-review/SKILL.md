# testspec-review：用例评审

## 职责

以独立评审视角，对已生成的测试用例进行交叉验证，检查覆盖度、命名规范、优先级分布、字段完整性、可执行性、可追溯性、冗余检测、预期结果质量、意图一致性、前置条件充分性、风险与边界覆盖、可维护性、回归价值、测试亮点等 14 个维度，输出结构化的评审报告（review-report.md），帮助团队识别用例质量问题并提供整改建议。

## 当前变更目录

参见 `../testspec-shared/common.md` 中的「当前变更目录定位规则」。

## 自主深度校准

> 替代 `--deep` flag 的手动控制。Agent 基于上下文信号自主决定检查深度。

### 深度信号检测

在执行评审前，自动检测以下信号来决定检查深度：

**复杂度信号（触发加深检查）：**
- 用例总数 > 50 条
- 涉及模块数 > 5 个
- 包含 3 种以上用例类型
- testcases.json 为 Strict 模式（有 tp_refs）

**风险信号（对风险区域加深检查）：**
- 上游产物包含 `risks_identified` 元数据 → 对这些风险对应的用例重点检查
- 上游 `material_quality` 为 `low` → 全面加深检查
- 上游 `open_questions` 非空 → 检查相关区域是否有覆盖缺口

**简单信号（可简化检查）：**
- 用例总数 < 20 条
- 只有 1-2 个模块
- 上游 `material_quality` 为 `high`

### 深度校准结果

- **标准深度**：执行所有 14 项检查，H3 仅对高风险 TP 全解释
- **加深深度**：自动等效于 --deep 模式（额外读取 requirements-analysis.md，H3 对所有 TP 全解释，H5 要求 ≥ 3 种类型）
- **向用户说明**：告知用户选择了什么深度、为什么

## 反模式识别（Agent 自检参照）

> 评审过程中，对照以下反模式自查评审报告本身的质量。

| 反模式 | 表现 | 正确做法 |
|--------|------|----------|
| **走形式检查** | 每项检查都"通过"但未真正审视内容 | 对每项检查提供具体的量化指标和证据 |
| **模糊建议** | 整改建议是"需要改进""建议优化" | 每条建议必须指出改哪个用例、改什么字段、改成什么 |
| **遗漏 ID** | 报告问题时没有列出具体的用例 ID | 每个问题必须关联到具体的 case_id 或 TP_ID |
| **比例至上** | 只看优先级/类型分布比例，不看内容合理性 | 比例是参考，核心是每个 P1 用例是否确实覆盖关键路径 |
| **忽视上游** | 评审时不参考上游分析和测试点的风险标注 | 必须消费上游 context 元数据，对风险区域加深检查 |
| **过度报告** | 把所有细微问题都标为 S1 阻断级 | 严格按严重级别体系分级，S1 只用于真正的阻断问题 |

## 输入策略

### 必读文件

- **testcases.json**：生成的测试用例 JSON 文件（位于变更目录根目录或 artifacts/ 下）
- **specs/testpoints.md**：测试点文档（用于覆盖度检查和意图一致性验证）

### 可选文件（deep 模式）

- **requirements-analysis.md**：需求分析文档（deep 模式下用于增强意图一致性和风险覆盖检查）

### 输入健康检查（Step 2.5）

在执行评审前，必须进行以下健康检查：

1. **testcases.json 存在性检查**：
   - 文件存在且非空
   - JSON 格式有效（可解析）
   - 包含 `testcases` 数组且数组非空

2. **testpoints.md 存在性检查**：
   - 文件存在且非空
   - 包含测试点标识（TP_ID）

3. **健康检查失败处理**：
   - 若任一必读文件缺失或无效，输出错误信息并终止评审
   - 提示用户先执行 testspec-generate 或 testspec-points

### 模式判定（Strict/Legacy）

根据 testcases.json 的 schema 判定评审模式：

- **Strict 模式**：
  - testcases.json 包含 `schema_version: 2`
  - 所有用例包含 `tp_refs` 字段且非空
  - 可进行精确的可追溯性检查和意图一致性验证

- **Legacy 模式**：
  - testcases.json 不包含 `schema_version` 或 `schema_version < 2`
  - 用例缺少 `tp_refs` 字段
  - 可追溯性检查降级为警告（⚠️ S2）
  - 意图一致性检查 fallback 到标题匹配，置信度下调

## 评审维度

### 规则检查（R1-R6）

#### R1: 测试点覆盖度

**检查目标**：验证测试用例是否充分覆盖测试点文档中的所有测试点。

**检查方法**：
1. 从 testpoints.md 提取所有 TP_ID
2. Strict 模式：统计 testcases.json 中所有用例的 `tp_refs` 字段，计算覆盖率
3. Legacy 模式：通过标题匹配推断覆盖情况（置信度较低）
4. 计算覆盖率 = 已覆盖 TP 数 / 总 TP 数 × 100%

**判定标准**：
- ✅ 通过：覆盖率 ≥ 95%
- ❌ 失败（S2）：覆盖率 < 95%

**输出**：
- 覆盖率百分比
- 未覆盖的 TP_ID 列表
- 每个未覆盖 TP 的影响说明和整改建议

---

#### R2: 命名契约

**检查目标**：验证用例标题是否符合三段式命名规范。

**检查方法**：
1. 遍历所有用例的 `title` 字段
2. 检查格式是否为 `{模块}_{功能点}_{场景}`（使用下划线分隔）
3. 验证 `{模块}` 是否与 `feature` 字段一致
4. 验证 `{模块}` 和 `{功能点}` 是否与 testpoints.md 中的标题层级严格一致

**判定标准**：
- ✅ 通过：所有用例标题符合命名契约
- ❌ 失败（S2）：存在违规用例

**输出**：
- 违规用例数
- 每个违规用例的 ID、当前标题、违规原因、建议标题

---

#### R3: 优先级分布

**检查目标**：验证用例优先级分布是否合理。

**检查方法**：
1. 统计冒烟用例、P1、P2、P3 用例数量
2. 计算各级别占比
3. 检查冒烟用例是否全部为 P1
4. 检查分布合理性（纯 P1 或纯 P3 是异常信号）

**判定标准**：
- ✅ 通过：
  - 至少存在 2 个优先级级别
  - 所有冒烟用例均为 P1
  - 分布符合业务风险特征（Agent 自主判断，无固定比例）
- ⚠️ 警告（S3）：
  - 只有单一优先级
  - P1 占比 > 80%（可能缺乏优先级区分）
  - P3 占比 > 50%（可能低估了核心功能的重要性）

**输出**：
- 各级别用例数和占比
- 分布异常的级别及建议调整方向

---

#### R4: 字段完整性

**检查目标**：验证用例必填字段是否完整。

**检查方法**：
1. 遍历所有用例
2. 检查必填字段：`id`、`title`、`steps`、`expected_result`、`priority`
3. 检查字段值是否非空（去除空白字符后长度 > 0）

**判定标准**：
- ✅ 通过：所有用例的必填字段均非空
- ❌ 失败（S1）：存在缺失或空字段

**输出**：
- 缺失字段的用例数
- 每个问题用例的 ID、缺失字段列表、整改建议

---

#### R5: 可执行性最小条件

**检查目标**：验证用例是否满足可执行的最小条件。

**检查方法**：
1. 检查 `steps` 和 `expected_result` 长度是否 ≥ 10 字符
2. 检查 `steps` 是否包含动作动词（点击、输入、选择、等待、查看、校验、打开、提交、确认、删除、修改、搜索、上传、下载）

**判定标准**：
- ✅ 通过：所有用例满足最小条件
- ❌ 失败（S1）：存在不满足条件的用例

**输出**：
- 不满足条件的用例数
- 每个问题用例的 ID、不满足的条件、整改建议

---

#### R6: 可追溯性

**检查目标**：验证用例与测试点的追溯关系是否完整。

**检查方法**：
- **Strict 模式**：
  1. 检查所有用例是否包含 `tp_refs` 字段
  2. 检查 `tp_refs` 是否非空且包含有效的 TP_ID
  3. 计算 tp_refs 映射率 = 有 tp_refs 的用例数 / 总用例数 × 100%
- **Legacy 模式**：
  1. 检测到缺少 `tp_refs` 字段
  2. 输出警告（⚠️ S2），提示重新生成用例以支持精确追溯

**判定标准**：
- **Strict 模式**：
  - ✅ 通过：tp_refs 映射率 = 100%
  - ❌ 失败（S1）：tp_refs 映射率 < 100%
- **Legacy 模式**：
  - ⚠️ 警告（S2）：无 tp_refs 字段，建议重新生成

**输出**：
- 追溯完整性百分比
- 缺少 tp_refs 的用例列表
- Legacy 模式下的升级建议

---

### 启发式检查（H1-H8）

#### H1: 冗余检测

**检查目标**：识别可能冗余的测试用例。

**检查方法**（两阶段）：
1. **候选生成阶段**：
   - 计算用例间的相似度（基于标题、步骤、预期结果）
   - 标题相似度 > 80% 或步骤相似度 > 70% 的用例对标记为候选
2. **AI 判断阶段**：
   - 对候选用例对，使用 LLM 判断是否真正冗余
   - 考虑测试场景、数据、验证点的差异

**判定标准**：
- ✅ 通过：无疑似冗余用例对
- ⚠️ 警告（S3）：存在疑似冗余用例对

**输出**：
- 疑似冗余用例对数
- 每对用例的 ID、相似度、冗余原因、合并建议

---

#### H2: 预期结果质量

**检查目标**：检测预期结果中的模糊表述。

**检查方法**：
1. 遍历所有用例的 `expected_result` 字段
2. 检测模糊词汇：正常、成功、正确、显示、提示（无具体内容）
3. 检查是否缺少具体的验证标准（数值、状态、文案）

**判定标准**：
- ✅ 通过：所有预期结果具体、可验证
- ⚠️ 警告（S3）：存在模糊表述

**输出**：
- 模糊表述用例数
- 每个问题用例的 ID、模糊表述内容、改进建议

---

#### H3: 意图一致性

**检查目标**：验证用例是否准确反映测试点的验证意图。

**检查方法**：
- **Strict 模式**：
  1. 按 TP_ID 分组用例（通过 `tp_refs` 字段）
  2. 对每个 TP，读取其验证要点
  3. 逐条评估用例与 TP 的一致性，三段评分：
     - **Action Match**（操作匹配）：用例步骤是否覆盖 TP 要求的操作
     - **Oracle Match**（预期匹配）：用例预期结果是否覆盖 TP 的验证点
     - **Scope Match**（范围匹配）：用例是否引入了 TP 之外的验证内容
  4. 综合评分：三项均达标 → 一致，否则 → 偏移
- **Legacy 模式**：
  1. Fallback 到标题匹配（通过标题前缀推断 TP）
  2. 置信度下调，评估结果标注为"低置信度"

**预算阀门**：
- 默认模式：对高风险 TP（P1、冒烟相关）全解释
- Deep 模式：对所有 TP 全解释

**判定标准**：
- ✅ 通过：所有用例与 TP 意图一致
- ❌ 失败（S1）：存在意图偏移的用例

**输出**：
- 意图一致性评估表（TP_ID、用例数、Action Match、Oracle Match、Scope Match、结论、证据/说明）
- 意图偏移的用例列表及整改建议

---

#### H4: 前置条件充分性

**检查目标**：验证用例前置条件是否充分、可执行。

**检查方法**（抽样策略）：
1. 每个模块抽样：
   - P1 用例：3 条
   - 冒烟用例：3 条
   - 随机用例：3 条
2. 对抽样用例，检查前置条件是否：
   - 明确具体（无模糊表述）
   - 可执行（有明确的准备步骤）
   - 完整（覆盖所有必要的前置状态）

**判定标准**：
- ✅ 通过：抽样用例的前置条件均充分
- ⚠️ 警告（S2）：存在前置条件不充分的用例

**输出**：
- 抽样数和不充分数
- 每个问题用例的 ID、前置条件问题、改进建议

---

#### H5: 风险与边界覆盖

**检查目标**：验证用例是否覆盖多种测试类型（正向、负向、边界、异常、冒烟），并量化各设计技术维度的覆盖率。

**检查方法**：
1. 统计各类型用例数量
2. 检查覆盖的类型数
3. **技术维度覆盖率量化**（当上游 requirements-analysis.md 中明确标注了对应分析内容时计算）：
   - **等价类覆盖率** = 已覆盖分区数 / analysis 中识别的总分区数 × 100%（从等价类分析表提取）
   - **边界值覆盖率** = 已测边界值数 / analysis 中识别的总边界值数 × 100%（从边界值列表提取）
   - **状态转换覆盖率** = 已测转换数 / analysis 中识别的总有效转换数 × 100%（从状态迁移列表提取）
   - 若 analysis 未包含对应分析内容，跳过该维度（不强制计算）

**判定标准**：
- ✅ 通过（默认模式）：覆盖 ≥ 2 种类型
- ⚠️ 警告（S2，默认模式）：覆盖 < 2 种类型
- ❌ 失败（S2，deep 模式）：覆盖 < 3 种类型
- 技术维度覆盖率：任一维度 < 80% 时标注 ⚠️ S2 并给出补充建议

**输出**：
- 覆盖的类型列表和各类型用例数
- 缺失的类型及补充建议
- 技术维度覆盖率表（仅在有数据时输出）：

| 设计技术 | 总项数 | 已覆盖 | 覆盖率 | 状态 |
|----------|--------|--------|--------|------|
| 等价类   | N      | M      | M/N%   | ✅/⚠️ |
| 边界值   | N      | M      | M/N%   | ✅/⚠️ |
| 状态转换 | N      | M      | M/N%   | ✅/⚠️ |

---

#### H6: 可维护性建议

**检查目标**：提供用例可维护性优化建议。

**检查方法**：
1. 基于 H1 冗余检测结果，提供合并建议
2. 检查用例粒度是否适中（步骤数 3-10 为宜）
3. 检查用例是否过于复杂（步骤数 > 15）

**判定标准**：
- ✅ 通过：无明显可维护性问题
- ⚠️ 警告（S3）：存在可优化项

**输出**：
- 建议数
- 每条建议的类型（冗余合并、粒度调整、复杂度简化）、涉及用例、具体建议

---

#### H7: 回归价值评估

**检查目标**：评估用例能否在真实缺陷发生时有效捕获问题，识别"仪式感用例"（为覆盖率而写但永远不会失败的用例）。

**为什么需要这项检查**：结构完整、命名规范的用例未必有实际发现缺陷的能力。一条用例的价值不在于它存在，而在于当代码出 bug 时它能变红。如果一条用例无论代码怎么改都会通过，它就是在消耗维护成本而非保护质量。

**检查方法**（抽样 + 反事实推理）：
1. 从以下维度各抽样 3 条用例（共约 9 条，可能有重叠）：
   - P1 用例
   - 冒烟用例
   - 上游 `risks_identified` 中标注的风险区域对应的用例
2. 对每条抽样用例，进行**反事实提问**：
   - "如果这个功能的核心逻辑出了 bug（如条件判断反转、边界值偏移、状态未更新），这条用例的步骤和预期结果能捕获到吗？"
   - "这条用例的预期结果是否足够具体，使得缺陷发生时结果会明确不匹配？"
3. 对每条抽样用例，进行 **Oracle 审查**（预期结果的判断依据是否可靠）：
   - **需求一致性**：预期结果是否有明确的需求条款支撑？若需求未定义该行为，预期结果可能是编造的
   - **历史一致性**：预期结果是否与系统已有版本的行为一致？若是新功能则不适用
   - **同类产品一致性**：预期结果是否符合同类产品的通行惯例？（如"删除前二次确认"是行业惯例，但需求未写时不能作为唯一判断依据）
   - **隐式预期**：预期结果是否依赖了未写明的假设？（如"返回首页"——首页是哪个页面？登录前和登录后的首页不同）

   Oracle 审查的核心问题：**如果让另一个测试人员执行这条用例，他能否仅凭预期结果判断测试是否通过？** 如果需要"猜"或"问人"才能判断，说明 oracle 不充分。
4. 识别以下"仪式感用例"特征：
   - 预期结果过于宽泛（如"页面正常显示"），任何实现都能通过
   - 步骤只验证 UI 存在性而非业务逻辑正确性（如"按钮存在"而非"点击后状态变更为 X"）
   - 前置条件和步骤与预期结果之间缺乏因果链（步骤做的事和验证的事无关）

**判定标准**：
- ✅ 通过：抽样用例均具备缺陷捕获能力
- ⚠️ 警告（S3）：存在 1-2 条仪式感用例
- ❌ 失败（S2）：存在 3 条以上仪式感用例，说明用例生成策略可能系统性地偏向"形式完整"而非"价值驱动"

**输出**：
- 抽样用例数及评估结论
- 每条仪式感用例的 ID、问题诊断（为什么它抓不到 bug）、改写建议（如何让预期结果更具缺陷敏感性）
- 若发现系统性问题，给出策略级建议（如"预期结果应包含具体的数据状态或业务规则断言，而非界面描述"）

---

#### H8: 测试价值与亮点识别

**检查目标**：识别用例集中做得好的部分和可复用的测试模式，为团队提供正面反馈和知识沉淀。

> 评审不应只挑毛病——发现好的模式并推广，和发现问题并修正同样重要。一份只有批评的评审报告会让团队失去改进动力。

**检查方法**：
1. 扫描全部用例，识别以下亮点：
   - **高质量用例标杆**：预期结果具体、步骤清晰、场景有价值的用例（推荐作为 testlib 标杆案例）
   - **可复用测试模式**：多个用例共享的、设计良好的前置条件模板或验证范式
   - **创意场景**：非标准方法（非等价类/边界值）产生的、有独特发现价值的测试场景
2. 评估整体测试策略的健康度：
   - 冒烟用例是否精准覆盖了核心业务闭环（质量而非数量）
   - 优先级分布是否反映了真实的业务风险梯度
   - 用例类型分布是否体现了有意识的测试设计（而非机械套模板）

**判定标准**：
- 此项无通过/失败判定——纯正面反馈
- 识别到标杆用例时，标注推荐入库 testlib

**输出**：
- 标杆用例列表（用例 ID、推荐理由、建议入库路径）
- 可复用模式列表（模式描述、适用场景、涉及用例 ID）
- 测试策略健康度评语（1-2 句总结）

---

## 严重级别体系

评审问题按严重程度分为三级：

- **S1 阻断级**：必须修复，否则用例不可执行或严重偏离测试意图
  - 示例：缺少必填字段、不满足可执行性最小条件、意图偏移、tp_refs 映射不完整（Strict 模式）
- **S2 重要级**：应当修复，影响用例质量和可维护性
  - 示例：命名违规、覆盖率不足、前置条件不充分、无 tp_refs（Legacy 模式）
- **S3 建议级**：可选优化，提升用例质量和可维护性
  - 示例：冗余用例、预期结果模糊、优先级分布不均、可维护性改进

## 执行步骤

1. **定位当前变更目录**（按 common.md 规则）
2. **读取必读文件**：
   - testcases.json（变更目录根目录或 artifacts/ 下）
   - specs/testpoints.md
3. **输入健康检查 + 模式判定**（Step 2.5）：
   - 检查 testcases.json 存在性、格式有效性、非空性
   - 检查 testpoints.md 存在性、非空性
   - 判定评审模式（Strict/Legacy）
   - 若健康检查失败，输出错误信息并终止
4. **执行规则检查**（R1-R6）：
   - 按顺序执行 6 项规则检查
   - 记录每项检查的结果、指标、问题列表
5. **执行启发式检查**（H1-H8）：
   - 按顺序执行 8 项启发式检查
   - 记录每项检查的结果、指标、问题列表
6. **生成评审报告**：
   - 使用 `review-report-template.md` 作为模板
   - 填充所有检查结果、问题列表、整改建议
   - 计算总体置信度（Strict 模式 + 所有检查通过 → 高置信度；Legacy 模式 → 中置信度；输入不完整 → 低置信度）
   - 输出 review-report.md 到变更目录根目录
7. **输出总结**：
   - 报告路径
   - 总评结论（通过/有问题）
   - S1 阻断级问题数
   - 评审模式（Strict/Legacy）
   - 置信度（高/中/低）

---

## 触发方式

- **`testspec-review`**（默认）：执行所有 14 项检查，H3 仅对高风险 TP 全解释，H5 要求 ≥ 2 种类型
- **`testspec-review --deep`**：额外读取 requirements-analysis.md，H3 对所有 TP 全解释，H5 要求 ≥ 3 种类型
- **自主加深**：即使用户未指定 --deep，当检测到复杂度/风险信号时自动等效 deep 模式（见「自主深度校准」）

---

## 产出

**review-report.md**：结构化评审报告，位于变更目录根目录。结构参见 `review-report-template.md`。

---

## 总体置信度计算

- **高置信度**：Strict 模式 + 所有必读文件完整 + 所有检查项可执行
- **中置信度**：Legacy 模式（无 tp_refs），H3/R6 结论可信度下降
- **低置信度**：输入文件不完整，部分检查项无法执行

## 反馈合成闭环

> 评审结果不止是报告，还应为上游 skill 提供结构化反馈，形成人工驱动的闭环。

### 给 generate 的反馈

在评审报告中，当发现以下问题时，生成可直接操作的反馈：

- **需要补充的用例**：明确指出补充哪个 TP 的哪种场景
- **需要修正的用例**：指出用例 ID、问题字段、修正建议
- **策略调整建议**：如冒烟占比偏高/偏低，建议调整方向

### 给 points 的反馈

- **TP 粒度问题**：某些 TP 过粗导致用例展开困难，建议拆分
- **TP 缺失**：发现用例覆盖缺口对应的测试点缺失

### 给 analysis 的反馈

- **需求缺口**：评审中发现的业务逻辑漏洞或需求不明确点
- **风险补充**：评审中发现但 analysis 未识别的风险

### 上下文播种

在 review-report.md 末尾，按 `context-protocol.md` 播种元数据：

```markdown
<!-- testspec-context
{
  "source_skill": "testspec-review",
  "risks_identified": ["<评审中新发现的风险>"],
  "feedback_for_generate": ["<给 generate 的结构化反馈>"],
  "feedback_for_points": ["<给 points 的结构化反馈>"],
  "feedback_for_analysis": ["<给 analysis 的结构化反馈>"]
}
-->
```

### 闭环流程

评审完成后，告知用户：
1. 评审报告路径
2. 如有 S1/S2 问题，建议重新运行 testspec-generate 修正
3. 如有上游缺口，建议回到 testspec-points 或 testspec-analysis 补充
4. 用户决定是否执行修正
