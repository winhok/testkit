# TestSpec 分析模式单一数据源

> 目标：为 `testspec-analysis` 提供稳定、可复用的分析模式定义。此文件只维护模式边界、关注点、问题准入标准与产物结构，不承载执行流程。

## 使用原则：自适应探测

> 替代固定的「6 选 N」模式路由。Agent 通过快速扫描→信号驱动→自主终止的流程，动态决定分析深度。

### 第一步：快速扫描（固定执行）

通读全部材料，标记所有信号。不预设使用哪个模式，只收集信号：

- 出现”待定””TBD””后续确认” → 记录 completeness 信号
- 预期结果描述为”正常””成功” → 记录 testability 信号
- 涉及第三方依赖、性能要求、加密 → 记录 feasibility 信号
- 出现”适当””合理””尽快”等模糊词 → 记录 clarity 信号
- 同一概念多种命名 → 记录 consistency 信号
- 多状态转换、条件组合 → 记录 logic 信号

### 第二步：信号驱动的深度探测（按需触发）

根据第一步的信号密度，自动启动对应分析模式：

- **信号数 ≥ 3**：启动该模式的完整分析
- **信号数 = 1-2**：启动该模式的轻量分析（只报告高优问题）
- **信号数 = 0**：跳过该模式

用户明确指定模式时，无论信号密度如何都执行该模式。

### 第三步：自主终止判断

- 所有高密度信号对应的模式都已执行 → 结束分析
- 发现需要更多材料才能继续 → 向用户说明缺什么，暂停该模式
- 分析过程中发现新信号（如 completeness 分析中发现一致性矛盾）→ 动态追加模式

### 兜底策略

如果快速扫描后所有模式信号均 < 3：默认执行 `completeness` + `testability` + `logic`。这意味着材料较简洁或较完善，用基础三件套覆盖即可。

### 分析原则

- 分析结论应稳定、可复核，不依赖具体实现方案
- 分析阶段只回答”要关注什么风险”，不输出测试步骤、具体测试数据或实现细节
- Agent 向用户说明：启动了哪些模式、为什么启动（检测到的信号）、跳过了哪些模式

## 通用输出契约

所有分析模式共享以下产物结构：

- `analysis_type`: 模式标识
- `summary`: 一句话说明最关键问题
- `issues`: 问题列表
- `strengths`: 现有材料中已明确、可复用的优点
- `recommendations`: 后续补充或澄清建议

每条 `issue` 至少包含：

- `severity`: `high` / `medium` / `low`
- `category`: 问题类别
- `description`: 缺失、歧义、矛盾或风险的具体描述
- `location`: 文档中的功能、章节或交互位置
- `suggestion`: 建议补充或澄清的内容

## completeness

- 名称：完整性分析
- 目标：识别会导致开发无法直接实现、反复确认或返工的信息缺失
- 推理触发信号：
  - 材料中频繁出现"待定""TBD""后续确认"
  - 功能描述只有名称没有细节
  - 缺少错误处理和异常流程的描述
- 关注视角：开发落地、测试验收、交互闭环、权限与异常分支
- 核心问题：
  - 哪些功能只有名称，没有可执行细节
  - 哪些交互反馈、跳转、错误提示未定义
  - 哪些输入约束、校验规则、权限规则缺失
  - 哪些并发、冲突、异常处理场景未说明
- 只报告：
  - 会导致开发返工
  - 会导致理解分歧
  - 会直接影响验收
- 不报告：
  - 锦上添花型建议
  - 与当前需求范围无关的最佳实践

## testability

- 名称：可测性分析
- 目标：识别导致测试用例无法直接编写、无法构造数据或无法判断通过失败的信息缺口
- 推理触发信号：
  - 预期结果描述为"正常""成功"等模糊词
  - 缺少明确的成功/失败判定标准
  - 测试数据准备条件不明确
- 关注视角：预期结果、边界条件、异常分支、测试数据准备
- 核心问题：
  - 操作后系统应表现什么，是否足以写断言
  - 成功/失败标准是否清晰
  - 边界值、范围、格式、数量限制是否定义
  - 异常输入、网络异常、重复操作是否有明确处理
  - 前置数据、账号、环境依赖是否可准备
- 只报告：
  - 导致无法编写测试用例
  - 导致无法判定测试结果
- 不报告：
  - 纯体验优化建议
  - 不影响断言设计的文案优化

## feasibility

- 名称：可行性分析
- 目标：识别可能导致项目延期、成本失控或技术风险暴露的需求点
- 推理触发信号：
  - 涉及第三方 API、外部服务或不稳定依赖
  - 提到高并发、大数据量、实时性要求
  - 包含加密、支付、敏感数据等安全敏感场景
- 关注视角：实现复杂度、性能瓶颈、安全风险、外部依赖、周期评估
- 核心问题：
  - 是否存在实现成本远高于表面描述的功能
  - 是否依赖高风险第三方服务或不稳定外部条件
  - 性能、并发、容量目标是否现实
  - 安全防护、限流、防刷、敏感数据保护是否缺位
  - 联调、测试、灰度与回滚成本是否被低估
- 只报告：
  - 会导致延期
  - 存在明确技术风险
  - 成本或周期被明显低估
- 不报告：
  - 泛泛而谈的“技术上可行/不可行”

## clarity

- 名称：清晰度分析
- 目标：识别会导致开发、测试、产品对同一需求产生不同理解的模糊表述
- 推理触发信号：
  - 出现"适当""合理""尽快""相关"等模糊词
  - 同一概念在不同段落用不同名称
  - 接口文档中参数含义未解释
- 关注视角：术语歧义、模糊词、信息断层、接口语义不清
- 核心问题：
  - 术语是否未定义或一词多义
  - 是否存在“适当”“合理”“尽快”等无法执行的模糊词
  - 同一句话是否可被多种方式理解
  - 接口入参、返回值、状态码、错误码是否歧义
- 只报告：
  - 会导致理解偏差
  - 需要反复确认
- 不报告：
  - 单纯风格润色建议

## consistency

- 名称：一致性分析
- 目标：识别文档内部前后矛盾、同一概念多种说法、规则定义冲突
- 推理触发信号：
  - 同一按钮/字段/状态在不同章节有不同名称
  - 流程图与文字描述的步骤顺序不一致
  - 同一规则在不同上下文中定义的值域不同
- 关注视角：术语一致、流程一致、规则一致、数据定义一致
- 核心问题：
  - 同一功能、按钮、字段、状态是否多种命名
  - 同一规则在不同章节是否冲突
  - 流程图、原型、文字描述是否对不上
  - 数据类型、长度、范围、状态定义是否前后不一致
- 只报告：
  - 确认存在冲突或不一致的问题
- 必须给出：
  - 冲突双方的位置或描述来源
- 不报告：
  - 单纯风格不统一

## logic

- 名称：逻辑分析
- 目标：识别业务流程走不通、状态转换不闭环、异常分支遗漏或规则组合冲突
- 推理触发信号：
  - 存在多状态转换的业务流程（订单、审批、工单等）
  - 涉及多角色协作的复杂交互
  - 包含条件组合或规则优先级的业务逻辑
- 关注视角：流程闭环、状态迁移、条件分支、异常恢复、并发处理
- 核心问题：
  - 主流程是否有明确起点和终点
  - 异常分支是否能回到正常流程
  - 状态是否存在不可达或不可退出
  - 规则组合后是否自相矛盾
  - 并发、重复提交、重试、回滚是否有处理逻辑
- 只报告：
  - 流程走不通
  - 逻辑有漏洞
  - 状态流转不完整
- 不报告：
  - 纯实现优化建议

## 模式选择参考（用户显式指定时的快捷路由）

- 用户说”缺什么””哪里不完整””开发拿到能不能做” → `completeness`
- 用户说”能不能直接写用例””怎么判断通过” → `testability`
- 用户说”技术风险””做不做得出””性能安全” → `feasibility`
- 用户说”哪里有歧义””看不懂””表述不清” → `clarity`
- 用户说”前后不一致””自相矛盾” → `consistency`
- 用户说”流程是否走得通””状态是否有漏洞” → `logic`

> 注意：以上路由仅在用户显式指定时使用。未指定时，Agent 按「自适应探测」流程自主决定。
