# Review 维度细则

## 使用说明

- 按 R1 → R6 → H1 → H8 顺序执行
- 每个检查项都要输出：`结果`、`关键指标`、`问题列表`
- 问题列表每条必须包含：`severity`、`case_id/TP_ID`、`影响`、`建议`

---

## R1 覆盖度

- 目标：校验 TP 覆盖率
- Strict：按 `tp_refs` 精确计算
- Legacy：按标题前缀推断（低置信度）
- 阈值：覆盖率 >= 95% 通过，否则 S2

## R2 命名契约

- 目标：校验 `{模块}_{功能点}_{场景}`
- 校验项：
  - 三段式格式
  - 标题模块与 `feature` 一致
  - 模块/功能点与 `testpoints.md` 层级一致
- 违规：S2

## R3 优先级分布

- 目标：校验优先级是否有区分
- 关键项：
  - 冒烟必须全部 P1
  - 单一优先级、P1 > 80%、P3 > 50% 视为异常信号
- 异常：S3（分布异常但不一定阻断）

## R4 字段完整性

- 必填：`id`、`title`、`steps`、`expected_result`、`priority`
- 任一缺失：S1

## R5 可执行性最小条件

- `steps` 和 `expected_result` 应具备最小可执行信息
- `steps` 至少包含动作动词（点击/输入/选择/提交等）
- 不满足：S1

## R6 可追溯性

- Strict：
  - 所有用例都要有非空 `tp_refs`
  - `tp_refs` 指向有效 TP_ID
  - 映射率必须 100%
  - 否则 S1
- Legacy：缺少 `tp_refs` 记 S2，并给出升级建议

---

## H1 冗余检测

- 两阶段：
  1. 相似度筛候选（标题/步骤）
  2. 语义复核是否真重复
- 疑似重复：S3

## H2 预期结果质量

- 检查模糊表达："正常"、"成功"、"正确" 等无判定标准句式
- 检查断言贫乏（尤其 API 场景）：
  - 仅校验 `status_code == 200`
  - 缺少业务字段断言或结构断言
- 模糊表达：S3
- 断言贫乏：S2

## H3 意图一致性

- Strict：按 TP 逐条做 Action/Oracle/Scope 三段评估
- Legacy：标题匹配 fallback，需标注低置信度
- 默认深度：仅高风险 TP 全解释
- 加深深度：所有 TP 全解释
- 意图偏移：S1

### 三段评估定义

- **Action Match**：用例步骤是否真正覆盖了 TP 要求验证的动作或输入条件
- **Oracle Match**：用例预期结果是否覆盖了 TP 的验证目标，而不是只验证表面现象
- **Scope Match**：用例是否引入了 TP 之外的大量验证内容，导致单条用例意图漂移

### 执行方法

1. 从 `testpoints.md` 读取 TP 的“验证要点”
2. 对照 `tp_refs` 找到对应的用例集合
3. 先看步骤是否落实了 TP 的核心动作或条件
4. 再看预期结果是否给出了可判定的业务结论
5. 最后检查是否掺入过多旁支验证，导致一条用例同时承担多个不相干目标

### 结论规则

- 三项都成立：判定一致
- `Action Match` 缺失：通常说明步骤没有真正覆盖 TP，要优先修正
- `Oracle Match` 缺失：通常说明预期结果过虚，抓不到真实缺陷
- `Scope Match` 缺失：通常说明 TP 过粗或用例塞入了多个验证意图

## H4 前置条件充分性

- 抽样建议（每模块）：
  - P1 3 条
  - 冒烟 3 条
  - 随机 3 条
- 前置条件需具体、可执行、完整
- 不充分：S2

## H5 风险与边界覆盖

- 统计类型覆盖：冒烟/正向/负向/边界/异常/其他
- 标准深度：至少覆盖 2 种类型
- 加深深度：至少覆盖 3 种类型
- 若 analysis 提供等价类/边界值/状态迁移清单，计算维度覆盖率：
  - 任一维度 < 80% 记 S2
- 类型覆盖不足：S2

## H6 可维护性建议

- 识别可合并冗余、粒度过粗/过细、超长步骤
- 默认 S3

## H7 回归价值评估

- 抽样：P1、冒烟、风险区域各 3 条（可重叠）
- 反事实提问：
  - 逻辑反转/边界偏移时能否变红
  - 预期结果是否足够精确
  - 是否仅验证“表面响应”而漏检业务副作用
- Oracle 审查：
  - 需求一致性
  - 隐式假设是否可判定
  - 写操作是否有状态落地验证
- 仪式感用例：
  - 1-2 条 -> S3
  - >=3 条 -> S2

### 反事实提问方法

- 如果核心条件判断被写反，这条用例会失败吗
- 如果边界值偏移 1 位，这条用例会失败吗
- 如果状态没有真正落库、只返回了“成功”，这条用例会失败吗
- 如果系统只给出页面提示但业务结果未变化，这条用例会失败吗

### Oracle 审查维度

- **需求一致性**：预期结果是否有明确需求依据；如果需求没定义，不要把臆测写成正确答案
- **历史一致性**：已有系统行为是否支持这个预期；若是新功能可跳过
- **同类产品一致性**：可作为辅助参考，但不能替代需求依据
- **隐式假设可判定性**：另一个测试人员只看预期结果能否独立判定通过/失败
- **业务副作用完整性**：创建、更新、删除、转账、审批等写操作，不能只看 HTTP/UI 表面响应，必要时要验证数据状态或后续可观察结果

### 仪式感用例识别

- 预期结果过宽，如“页面正常显示”“操作成功”
- 步骤只检查控件存在，不验证业务状态变化
- 前置条件、步骤、预期之间没有明确因果链
- 写操作只验证状态码或提示文案，不验证落库、副作用或后续状态
- 敏感场景未检查越权、数据泄露或关键字段遗漏

## H8 亮点识别

- 标杆用例（可沉淀 testlib）
- 可复用模式（前置条件模板、断言模板、场景模板）
- 输出 1-2 句策略健康度结论
- H8 仅正向反馈，不判失败

- `整改建议清单` 必须按 S1/S2/S3 分层
