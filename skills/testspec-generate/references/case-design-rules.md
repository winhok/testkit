## 设计方法

按测试点特征选择方法，不机械套用：

1. 等价类划分：将输入或条件划分为有效类与无效类，每类至少 1 个用例
2. 边界值分析：覆盖最小值、最大值、临界值、空值、最大/最小 +/- 1
3. 判定表驱动：多条件组合时列出条件、动作、预期，覆盖关键组合
4. Pairwise：多参数多取值组合爆炸时，确保所有参数两两组合至少覆盖一次
5. 场景法：按用户实际使用路径覆盖正常、异常、边界场景
6. 错误推测法：补充网络中断、并发操作、重复提交、安全风险等经验场景
7. 状态迁移法：状态机明确时覆盖合法和非法状态转换

## 创意测试探索

对中等及以上复杂度 TP 触发，简单 TP 跳过。问四个问题：

1. 真实用户最可能的非典型操作是什么？
2. 不同功能模块之间是否有意外交叉影响？
3. 两个操作几乎同时发生会怎样？
4. 除了数值边界，是否存在语义边界？

有价值的场景直接生成用例，标注为 `异常` 或 `边界`；没有好场景时不凑数。在 testcases.json 的 `_context` 中记录：

```json
{
  "creative_scenarios_added": 3,
  "creative_sources": ["用户真实行为", "跨场景组合"]
}
```

## 字段规范

用例至少包含：
- `id`：`{需求名称}_YYYYMMDD{SEQ}`，SEQ 四位递增；publish 会结合 testlib 状态做增量管理，不假设跨变更稳定
- `title`：`{模块}_{功能点}_{测试场景}`，前两段必须来自 testpoints.md 标题层级
- `feature`：必须等于 `{模块}`，用于 XMind 分组
- `type`：冒烟 / 正向 / 负向 / 边界 / 异常 / 其他
- `preconditions`：编号换行，如 `1、条件一\n2、条件二`
- `steps`：编号换行，动作清晰并包含关键测试数据
- `expected_result`：编号换行，具体到可验证状态、数值、文案或系统行为
- `priority`：P1 / P2 / P3
- `tp_refs`：至少 1 个当前变更内 TP_ID

JSON 字符串中的双引号和反斜杠必须合法转义；推荐用中文引号「」包裹 UI 文案，减少转义风险。

## 粒度控制

复杂度自适应生成：
- 简单 TP：2-4 条用例
- 中等 TP：5-8 条用例
- 复杂 TP：8-15 条用例

拆分信号：
- 多个独立验证维度
- 不同验证点失败影响不同
- 需要完全不同的测试数据
- 同一功能的不同验证点有不同优先级

合并信号：
- 验证点高度相关，属于同一原子操作
- 操作步骤完全相同，只是观察角度不同
- 某些验证点必须在同一操作流程中完成
- 拆分会造成用例膨胀且降低执行效率

## 反模式

| 反模式 | 表现 | 正确做法 |
|--------|------|----------|
| 复制粘贴预期 | 所有预期都是"操作成功""正常显示" | 每条预期写清具体状态、数值、文案或系统行为 |
| 过度前置条件 | 前置条件写大量公共环境配置 | 只写与本用例差异相关的条件 |
| 伪边界 | 只是换数值，没有新验证逻辑 | 边界用例必须触发通过/拒绝/提示等不同系统行为 |
| 机械展开 | 每个 TP 都展开相同数量用例 | 按复杂度决定 2-4、5-8、8-15 |
| 场景名无信息 | 标题为"测试1""场景A" | 场景名表达具体测试意图 |
| 孤立用例 | `tp_refs` 为空 | 每条用例至少关联 1 个 TP_ID |
