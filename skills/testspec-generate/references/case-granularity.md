# 用例粒度控制

## 复杂度自适应生成

> 替代固定比例指导。Agent 根据每个测试点的复杂度自主决定用例数量，而非套用全局比例。

Agent 评估每个测试点的复杂度后，决定展开深度：

- **简单**（单一输入、明确规则、无状态变化）→ 2-4 条用例
- **中等**（多输入组合、有状态转换、涉及权限）→ 5-8 条用例
- **复杂**（跨模块交互、复杂业务规则、多角色协作）→ 8-15 条用例

复杂度判断信号：

- 测试点的验证要点数量（1 个 = 简单，2-3 个 = 中等，4+ 个 = 复杂）
- 上游标注的风险等级（risks_identified 中出现该模块 → 加深）
- 涉及的条件组合数（条件越多 → 越复杂）

**优先级分布由 Agent 根据风险判断，不设固定比例**：

- 核心正向路径、阻塞性功能 = P1
- 常规业务功能、重要边界 = P2
- 低频功能、边缘探索、体验类 = P3
- 冒烟用例从 P1 中识别，标准：失败会阻塞大部分功能的最短业务闭环

## 何时拆分

- 多个独立验证维度（数据准确性、界面显示、性能时效等）
- 不同验证点失败时影响不同
- 拆分后能更精确定位问题
- 需要完全不同的测试数据
- 同一功能的不同验证点有不同优先级

## 何时合并

- 验证点高度相关，属于同一原子操作的不同表现
- 操作步骤完全相同，只是观察角度不同
- 某些验证点必须在同一操作流程中完成
- 拆分导致用例数量过于庞大
