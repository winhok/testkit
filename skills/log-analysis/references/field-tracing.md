# Field Tracing

用于字段为空、丢失、错值、跨系统不一致、入参出参不一致、数据库和接口结果不一致等问题。

## 核心原则

定位字段在链路中的最早异常点。不要只解释下游如何处理空值，也不要把“日志没打印”当成“服务没收到”。

## 字段状态定义

| 状态 | 含义 | 输出要求 |
|------|------|----------|
| 有值 | 日志明确出现非空值 | 引用首次出现位置 |
| 空值 | 日志明确出现 null、空串、空数组、缺省值或 SQL 参数为空 | 引用字段和上下文 |
| 未打印 | 当前日志没有该字段 | 明确“不能证明是否收到” |
| 名称变化 | 同一概念在不同层使用不同字段名 | 列出映射依据 |
| 来源不同 | 用户以为来自 A，日志显示来自 B | 说明实际数据源 |
| 被覆盖 | 上游有值，下游被置空或替换 | 找到覆盖发生的最早位置 |

## 通用追踪顺序

1. 入口：HTTP header/query/body、RPC 入参、消息 payload、任务参数。
2. 边界层：网关、拦截器、中间件、参数绑定、鉴权上下文。
3. 服务层：DTO/VO/model 映射、默认值、校验、分支逻辑。
4. 依赖调用：HTTP/RPC/SDK 请求体和响应体。
5. 存储层：SQL 参数、查询结果、缓存 key/value、写入结果。
6. 事件层：消息发布、消费、重试、死信、延迟队列。
7. 输出层：响应组装、序列化、脱敏、过滤、降级兜底。

## 字段别名处理

先按概念整理同义字段，不要硬编码特定业务名。常见类别：

| 类别 | 命名规律 |
|------|----------|
| 主体标识 | 执行动作或拥有资源的主体：`*Id` / `*No` / `*Key` / `*Code` / `*Ref` |
| 终端标识 | 发起请求的客户端、设备、会话、安装实例或环境标识 |
| 认证上下文 | token、session、credential、claim、principal 等身份上下文 |
| 业务对象标识 | 被操作对象、资源、记录、任务、消息或批次的标识 |
| 作用域字段 | 租户、组织、项目、应用、环境、地域等隔离维度 |
| 属性字段 | 名称、联系方式、展示资料、配置项等非主键属性 |
| 状态字段 | 状态、结果、类型、开关、删除标记、错误码等流程控制字段 |

只有在日志证据能支持时，才把两个字段视为同一概念。证据不足时写“疑似同义，需要代码或接口定义确认”。

## 常见根因类别

以下类别只作为候选解释。必须先找到日志证据支持对应跳点，不能默认把字段异常归因到某一层。

- 客户端或上游未传。
- 网关、中间件或参数绑定丢字段。
- DTO/model 映射漏字段或命名不一致。
- 鉴权主体和业务主体不是同一个实体。
- 查询了错误的数据源、租户、环境、表、缓存 key 或实体类型。
- 数据库或缓存本身无值。
- 下游返回空值，上游原样透传。
- 响应组装、脱敏、过滤、默认值逻辑覆盖了字段。
- 异步消息 payload 缺字段或使用旧 schema。

## 输出要求

必须包含：

- 字段别名：本次追踪的字段及疑似等价命名。
- 逐跳状态：每一跳是有值、空值、未打印、名称变化、来源不同还是被覆盖。
- 最早异常点：第一处可见异常或第一处无法证明的断点。
- 不能证明的地方：哪些服务、日志或字段没有覆盖。
- 下一步：最小 grep、SQL 或代码检查方向。
