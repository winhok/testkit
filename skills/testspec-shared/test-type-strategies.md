# TestSpec 测试类型策略单一数据源

> 目标：为 `testspec-generate` 提供稳定的测试类型策略映射。此文件定义”按什么策略展开用例”，不定义最终导出格式。

## 使用原则（Agentic 选择）

> Agent 不需要使用所有类型。根据需求材料中的信号自主判断哪些类型与当前被测对象相关，跳过无信号的类型。

- `testspec-generate` 在推理阶段扫描需求材料，检测每种类型的触发信号
- **有信号** → 纳入本次生成策略
- **无信号** → 跳过，不生成该类型的用例
- 用户显式指定类型时，无论信号如何都执行
- 未指定且无明显信号时，默认 `functional`
- 本文件只约束覆盖重点、设计技术和粒度，不改变 `testcases.json`、Excel、XMind 的字段结构

## smoke

- 名称：冒烟测试
- 目标：最小化验证核心主流程是否可用
- 推理触发信号：
  - 用户提到"快速验证""冒烟""基本可用"
  - 材料中有明确的核心业务闭环
  - 上线前紧急测试场景
- 设计策略：
  - 每个核心功能点最多展开 1-2 条用例
  - 优先覆盖最短业务闭环
  - 只保留阻塞性强、失败影响大的场景
- 推荐技术：
  - 主流程场景法
  - 关键节点错误推测
- 输出倾向：
  - 优先生成 `type=冒烟`
  - 必须为 `P1`

## functional

- 名称：功能测试
- 目标：验证功能正确性与主要业务路径
- 推理触发信号：
  - 未指定具体测试类型（默认策略）
  - 用户提到"功能测试""验证功能""完整测试"
  - 新功能上线需要全面验证
- 设计策略：
  - 覆盖有效等价类与关键无效等价类
  - 优先覆盖主路径、核心分支、关键规则
  - 不为追求全面而无限膨胀用例数
- 推荐技术：
  - 等价类划分
  - 场景法
  - 判定表
- 输出倾向：
  - 优先生成 `type=正向`，必要时补充 `负向`

## boundary

- 名称：边界测试
- 目标：验证数值、长度、容量、日期时间等边界条件处理
- 推理触发信号：
  - 材料中有明确的数值范围、长度限制、容量上限
  - 涉及表单输入、文件上传、数据导入等有约束的场景
  - 上游分析标注了边界值相关的风险
- 设计策略：
  - 优先覆盖边界值、边界值±1、空值、满值、典型值
  - 同一验证意图的数据点可以拆成多个独立用例
- 推荐技术：
  - 边界值分析
  - 等价类划分
- 输出倾向：
  - 优先生成 `type=边界`

## exception

- 名称：异常测试
- 目标：验证系统异常处理能力和错误反馈质量
- 推理触发信号：
  - 涉及网络请求、外部服务调用
  - 有支付、数据修改等需要事务一致性的操作
  - 上游分析标注了异常处理缺失的风险
- 设计策略：
  - 覆盖异常输入、重复提交、超时、网络异常、服务异常、并发冲突
  - 关注错误提示、回滚、幂等、数据一致性
- 推荐技术：
  - 错误推测
  - 异常场景法
  - 状态迁移
- 输出倾向：
  - 根据场景落到 `type=负向` 或 `type=异常`

## permission

- 名称：权限测试
- 目标：验证角色访问控制、功能入口暴露和数据隔离
- 推理触发信号：
  - 材料中有角色定义或权限矩阵
  - 涉及多租户、数据隔离
  - 上游分析标注了权限交叉或越权风险
- 设计策略：
  - 明确角色矩阵
  - 同时覆盖有权限、无权限、越权三类场景
  - 优先覆盖高风险操作和敏感数据访问
- 推荐技术：
  - 权限矩阵
  - 场景法
- 输出倾向：
  - 通常落到 `type=负向` 或 `type=异常`

## security

- 名称：安全测试
- 目标：验证基础安全防护机制是否缺失
- 推理触发信号：
  - 涉及用户认证、会话管理、密码处理
  - 有用户输入直接展示的场景（XSS 风险）
  - 涉及敏感数据传输或存储
- 设计策略：
  - 聚焦输入校验、输出编码、认证鉴权、敏感信息保护、会话管理
  - 只覆盖需求材料中能合理推导出的安全场景
- 推荐技术：
  - 错误推测
  - 风险驱动场景法
- 输出倾向：
  - 常落到 `type=异常` 或 `type=负向`

## compatibility

- 名称：兼容性测试
- 目标：验证系统在不同设备、浏览器、系统或版本中的表现一致性
- 推理触发信号：
  - 材料中指定了目标浏览器或设备列表
  - 涉及响应式布局、移动端适配
  - 有新旧版本并行或数据迁移场景
- 设计策略：
  - 先从需求中提取目标环境矩阵
  - 同一功能在不同环境中生成独立用例
  - 关注显示、交互、上传下载、输入法、布局、版本差异
- 推荐技术：
  - 环境矩阵
  - 场景法
- 输出倾向：
  - `type` 可保持业务类型；重点通过 `feature/title/steps` 区分环境

## usability

- 名称：易用性测试
- 目标：验证用户操作的直觉性、效率和容错性，确保交互体验符合预期
- 推理触发信号：
  - 材料中有交互流程图、原型图或 UI 设计稿
  - 涉及表单填写、多步骤向导、拖拽操作等复杂交互
  - 提到用户反馈、操作引导、帮助提示
  - ToC 产品或面向非技术用户的功能
- 设计策略：
  - 验证操作路径的最短性和直觉性
  - 检查错误提示的清晰度和可操作性（告诉用户"怎么修"而非仅"出错了"）
  - 验证空状态、加载状态、异常状态的用户引导
  - 关注键盘导航、Tab 顺序、焦点管理
  - 验证撤销/回退操作的可用性
- 推荐技术：
  - 用户旅程分析
  - 启发式评估（Nielsen 10 原则）
  - 场景法
- 输出倾向：
  - 常落到 `type=正向` 或 `type=其他`

## performance

- 名称：性能测试
- 目标：验证系统在负载下的响应时间、吞吐量和资源消耗是否满足需求指标
- 推理触发信号：
  - 材料中有明确的性能指标（响应时间 < Ns、支持 N 并发）
  - 涉及列表加载、文件上传/下载、批量操作、大数据量展示
  - 提到高并发、峰值流量、秒杀、抢购等场景
  - 上游分析标注了性能相关风险
- 设计策略：
  - 基于需求中的性能指标设计验证用例
  - 区分单用户响应时间和多用户并发场景
  - 覆盖正常负载、峰值负载、持续负载（耐久性）
  - 关注关键资源瓶颈（CPU、内存、网络、数据库连接）
- 推荐技术：
  - 负载建模
  - 边界值分析（并发数上限、数据量上限）
  - 基准对比
- 输出倾向：
  - 常落到 `type=其他`；预期结果必须包含量化指标

## reliability

- 名称：可靠性测试（含稳定性）
- 目标：验证系统在异常条件和长时间运行下的稳定性、容错能力和恢复能力
- 推理触发信号：
  - 涉及 7×24 运行的后台服务或关键业务系统
  - 材料中提到故障恢复、自动重连、断点续传
  - 涉及数据持久化、事务一致性
  - 有定时任务、消息队列消费、长连接等长时间运行组件
  - 上游分析标注了可靠性/稳定性风险
- 设计策略：
  - 验证异常恢复能力（网络断开恢复、服务重启后状态保持）
  - 验证长时间运行下的资源泄漏（内存、连接、文件句柄）
  - 验证数据一致性保障（事务回滚、幂等性、并发写入）
  - 关注降级策略和熔断机制
- 推荐技术：
  - 故障注入
  - 耐久性测试
  - 混沌工程场景
- 输出倾向：
  - 常落到 `type=异常` 或 `type=其他`

## tracking

- 名称：埋点测试
- 目标：验证用户行为数据采集的完整性、准确性和时效性
- 推理触发信号：
  - 材料中有埋点文档、事件列表或数据采集需求
  - 涉及用户行为分析、AB 测试、转化率统计
  - 提到 PV/UV、点击事件、曝光事件、停留时长
  - 产品明确要求验证数据上报
- 设计策略：
  - 逐事件验证：触发条件、上报参数、上报时机
  - 验证参数完整性（事件名、属性字段、用户标识）
  - 验证去重逻辑（重复曝光、快速点击）
  - 验证异常场景下的数据不丢失（弱网、页面切换、App 退后台）
- 推荐技术：
  - 埋点矩阵（事件 × 页面 × 触发条件）
  - 场景法
- 输出倾向：
  - `type=正向` 或 `type=其他`；步骤中需包含"检查上报数据"的验证动作

## maintainability

- 名称：可维护性测试
- 目标：验证系统的可配置性、可监控性和可运维性
- 推理触发信号：
  - 涉及后台配置项、运营开关、功能灰度
  - 材料中提到日志、监控、告警、运维操作
  - 有版本升级、数据迁移、配置热更新需求
- 设计策略：
  - 验证配置项修改后的即时生效或重启生效
  - 验证日志输出的关键信息完整性（请求ID、时间戳、错误堆栈）
  - 验证监控指标的准确性和告警触发条件
  - 验证数据备份恢复流程的可执行性
- 推荐技术：
  - 场景法
  - 检查清单
- 输出倾向：
  - `type=正向` 或 `type=其他`

## portability

- 名称：可移植性测试
- 目标：验证系统在不同运行环境间迁移的能力
- 推理触发信号：
  - 涉及多云部署、私有化交付、本地化部署
  - 材料中提到 Docker 化、容器编排、环境无关性
  - 有跨数据库、跨操作系统的适配需求
- 设计策略：
  - 验证在目标环境上的安装和配置流程
  - 验证数据导入/导出的完整性
  - 验证环境变量和配置文件的适配
- 推荐技术：
  - 环境矩阵
  - 安装部署场景法
- 输出倾向：
  - `type=正向` 或 `type=其他`

## effect_ai

- 名称：效果测试（AI/算法/资源类）
- 目标：验证 AI 模型、推荐算法、搜索排序等输出结果的准确性和效果指标
- 推理触发信号：
  - 涉及机器学习模型、推荐系统、智能排序、NLP 处理
  - 材料中有准确率、召回率、F1 等效果指标
  - 涉及内容审核、智能客服、OCR、语音识别等 AI 能力
  - 提到模型版本切换、AB 测试效果对比
- 设计策略：
  - 基于效果指标设计验证数据集（正例、负例、边界案例）
  - 验证模型输出的一致性和稳定性
  - 验证 bad case 的处理和兜底策略
  - 关注模型更新后的回归验证
- 推荐技术：
  - 数据驱动测试
  - 基准对比
  - 回归测试
- 输出倾向：
  - `type=正向` 或 `type=其他`；预期结果需包含效果指标

## effect_hardware

- 名称：效果测试（硬件/器件类）
- 目标：验证硬件交互、传感器数据采集、设备控制等场景的效果和稳定性
- 推理触发信号：
  - 涉及 IoT 设备、传感器、摄像头、打印机等硬件交互
  - 材料中有硬件协议、固件版本、设备型号适配需求
  - 涉及蓝牙、串口、USB、NFC 等通信方式
  - 提到设备绑定、配网、固件升级
- 设计策略：
  - 验证设备连接/断开的状态管理
  - 验证数据采集的精度和频率
  - 验证设备控制指令的响应时间和成功率
  - 验证多设备并发和设备切换场景
- 推荐技术：
  - 状态迁移
  - 设备矩阵
  - 错误推测
- 输出倾向：
  - 根据场景落到 `type=正向`、`type=异常` 或 `type=其他`

## 多类型组合规则

- `functional + boundary`：先保证正向主路径，再补关键边界
- `functional + exception`：先保主路径，再覆盖关键失败路径
- `smoke + functional`：冒烟只保最小闭环；其余核心场景落功能测试
- `permission + security`：先覆盖权限矩阵，再补高风险安全场景
- `functional + usability`：功能验证后补交互体验和引导检查
- `functional + performance`：先验证功能正确性，再叠加性能指标验证
- `functional + tracking`：功能用例中嵌入埋点验证步骤，或单独生成埋点用例
- `functional + reliability`：先正向功能，再补故障恢复和长时间运行场景
- `effect_ai + functional`：先验证基础功能可用，再叠加效果指标验证

## 默认回退

- 未识别类型时，回退为 `functional`
- Agent 根据材料信号自主组合类型，不需要用户显式指定所有类型
