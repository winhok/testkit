# 需求审问闭环

## 目录

- 触发条件
- 审问维度
- 输出格式
- 回答合并

## 触发条件

核心判断：这个问题是否影响测试用例设计或测试结果判断。是则记录，否則跳过。

触发更深入审问的信号：
- 材料中出现"待定""TBD""后续确认"
- 功能描述缺少异常处理路径
- 预期结果无法量化验证
- 业务规则存在歧义或矛盾

## 审问维度

1. 因果性：业务规则、权限前提、触发条件是否明确
2. 边界定义：数值范围、长度限制、容量上限是否定义
3. 异常场景：失败处理、重试、回滚、超时是否说明
4. 交互细节：提示文案、反馈方式、空状态、加载状态是否定义
5. 数据状态：字段定义、状态流转、约束条件是否完整
6. 性能兼容：响应时间、并发能力、兼容范围是否有要求

过滤规则：
- 只记录影响测试的问题，不记录文档风格、排版、锦上添花建议
- 每个问题指向一个具体可澄清项，避免"建议补充更多细节"

## 输出格式

在 `requirements-analysis.md` 的「主要问题」之后增加：

```markdown
## 需求审问清单

### 问题 1 [因果性 · 高优]

- **位置**：<需求文档中的具体位置>
- **问题**：<缺失/矛盾/模糊的具体描述>
- **测试影响**：<如果不澄清，哪些测试无法设计>
- **建议回答方向**：<对产品/开发的建议>
```

## 回答合并

当用户提供产品/开发回答、会议纪要，或直接修改「产品回答」字段时，进入合并模式：

1. 将回答与审问清单逐条匹配
2. 在对应问题下追加「产品回答」，标记为已澄清
3. 判断回答是否改变原有风险判断或分析结论
4. 将新规则、约束、边界值合并到「功能模块拆解」对应模块
5. 更新上下文元数据：
   - `open_questions` 移除已澄清项
   - `risks_identified` 根据回答新增或降级
   - `material_quality` 在多数问题澄清后上调

完成后告知用户：已合并多少回答、更新了哪些模块、仍有多少问题待确认、是否建议进入 testspec-points。
